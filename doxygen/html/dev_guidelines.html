Guidelines

  1. Code of Conduct
    • My project adheres to the Contributor Covenant code of conduct. All contributors are expected to uphold this code when interacting with the community and contributing to the project.
  2. How to Contribute
    • Reporting Issues: To report a bug or request a feature, please use GitHub isssue tracker. Before submitting an issue, research existing ones to avoid duplication.
    • Submitting Changes: Fork the repository and create new branch for your changes. Ensure commits include clear, descriptive messages.
    • Design: Improved designs of the webapp are more than welcome. If you want to contribute a design, please open an issue with clear label, so that I know this is a design contribution, or alternatively send me an email with the designs.
    • Code Reviews: All pull requests must pass code review. I provide feedback and approve changes before merging. You can learn more about this under the 'Code Review Process' page under the 'Contribution' section.
    • More Ways:You can find more ways how to contribute on the home page of this documentation website.
  3. Code Standards
    • Basic Standards: Follow basic conding standards for PHP, ensuring consistent and readable code. Use clear and descriptive variable names, proper indentation, and mainain a clean and oarganized code structure.
    • Documentation: Document all new code and include comment blocks or inline comments for complex logic. Update the project documentation as needed. You can learn more about updating documentation on the 'Updating Documentation' section under the 'Contribution' section.
  4. Testing Guidelines

    5. While testing is optional for contributors, it's highly recomended to perform through testing of your changes before submitting them for review. This ensures that your contributions meet the project's quality standards and minimizes the risk of introducing issues into the codebase. I will personally conduct comprehensive testing on project, but your effors in testing your changes will greatly contribute to the overall stability and reliability of the application. Please consider running the appropiate tests related to your contributions before finalizing your changes for submission.

    • Unit Tests: Write unit tests for new code and ensure existing tests pass. Aim for comprehensive test coverage.
    • Functionality Testing: Ensure that all features and functionalities work as expected.
    • Database Testing: Test all SQL querries to ensure they retrieve and update data accurately.
    • Component Testing: Test individual components to ensure they function correctly.
    • Usability Testing: Test the usability of the website, by filling out the provided checklist, or by making your own form.
  5. Versioning
    • Branch Strategy: Pawfect Care website follows a GitFlow branching model. Feature branches should be prefixed with 'feature/' and bug fixes with 'fix/'. Please create feature branches off the 'develop' branch and submit pull request targeting 'develop' for non-emergency changes
    • Release Process: Major versions are managed through release branches. For each major release, a new release branch is created from 'develop'. Version tags (e.g.v1.0.0) are used to mark stable release. Minor changes and bug fixes are integrateed into the 'develop' branch and eventually merged into release branches for deployment.
  6. License
    • Licensing: This project is licensed under the AGPL-3.0 license. All contributors are encouraged to share their changes with the community through pull requests to foster collaboration and improvement of the project. By contributing, you agree to license your contributions under the same terms.
